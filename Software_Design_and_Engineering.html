Artifact Description

The artifact I selected for this category is a Thermostat Prototype originally developed in my CS-350: Emerging Systems Architectures and Technologies. This project uses a Raspberry Pi integrated with a temperature sensor, an LCD screen, and LED indicators to simulate a climate control system. The software manages three states—Off, Heat, and Cool—and transmits status updates to a remote server via serial communication.

Justification for Inclusion

I selected this artifact because it serves as a bridge between physical hardware and complex software logic. By enhancing this project, I demonstrated growth in:

  • Architectural Design: Transitioning from a monolithic script to a modular, class-based structure.
  • Reliability: Implementing defensive programming to ensure sensor or display failures do not crash the primary control logic.
  • Maintainability: Centralizing hardware configurations to allow for rapid updates to physical pin mappings without altering the core logic.

The Enhancement: Modularization and Refactoring

My primary goal was to refactor the software into a "Toolbox" architecture. In the original version, the code was a linear list of instructions where global variables and hardware logic were tightly coupled. The enhanced version decouples these responsibilities into dedicated classes.

1. Centralized Configuration

I implemented a ‘CONFIG’ dictionary to manage all hardware pin assignments and thresholds in a single location. This follows industry best practices for maintainability.

2. Decoupled Hardware Logic

I created the ‘HardwareToolbox’ class to act as an abstraction layer between the physical sensors and the thermostat's brain. This ensures that the main controller doesn't need to know how the sensor works, only what the temperature is.

def read_temperature_f(self):
    """
    Reads from sensor and converts to Fahrenheit.
    Includes defensive checks to handle sensor detachment.
    """
    try:
        if self.thSensor:
            t_celsius = self.thSensor.temperature
            t_fahrenheit = ((9 / 5) * t_celsius) + 32
            
            # Cache the value for stability
            self.last_known_temp = round(t_fahrenheit, 1)
            return self.last_known_temp
        
        return self.last_known_temp
        
    except Exception as e:
        print(f"Error reading sensor: {e}")
        return self.last_known_temp

3. Professional Communication and Logic

The main ThermostatController now uses a non-blocking timing loop instead of multiple threads for simple display updates, making the execution flow easier to debug and more stable.

Growth & Technical Process

Throughout the enhancement process, I deepened my understanding of the separation of concerns principle. By isolating the LCD logic from the temperature-sensing logic into discrete class "toolboxes," I engineered a more resilient system. This modularity ensures that a peripheral failure, such as a visual glitch on the LCD, does not interfere with the core thermostat functions—maintaining the critical safety and regulation of the environment.

Overcoming Challenges

A significant technical challenge was managing the timing of serial server updates and display refreshes without relying on the Thread library. While threading can handle concurrent tasks, it often introduces complex race conditions and resource contention in embedded environments. I overcame this by implementing a non-blocking timestamp comparison loop. This approach allows the processor to remain highly responsive to physical button interruptions while precisely executing periodic tasks like data logging and screen updates.
